# 招行网银自动充值测试文档

## 1. 文档目的

本文档用于指导测试人员使用招商银行测试网银完成转账审批测试，并配合系统验证自动充值能力。

本次测试目标：

1. 配置招行测试环境访问能力；
2. 使用两个网银用户模拟“经办 + 审批”转账流程；
3. 使用第二个结算户向第一个结算户转账，制造系统可识别的收款流水；
4. 验证系统能够自动查询流水、匹配回单并完成充值。

## 2. 自动充值说明

### 2.1 自动充值原理

自动充值的核心思路是：测试人员先在系统中创建充值订单，再通过测试网银完成转账。银行入账后，系统会自动识别这笔收款，并尝试与充值订单进行匹配。

测试时需要重点关注以下匹配条件：

1. 收款账户必须是 `755915680110101`；
2. 订单创建时间必须早于实际转账时间；
3. 实际转账金额必须与订单金额一致；
4. 付款方名称必须与测试企业名称一致；
5. 同一笔流水只能匹配一笔订单，不能重复使用。

当以上条件满足时，系统应自动完成充值，无需人工确认。

### 2.2 兜底处理

如果自动识别未通过，则本次充值不应直接失败结束，而是继续按原来的人工确认流程处理。

人工确认仅作为自动识别失败时的补充处理方式，校验标准不应降低，也不应因为转人工而放宽。

人工确认时，仍应重点校验以下条件：

1. 收款账户必须是 `755915680110101`；
2. 订单创建时间必须早于实际转账时间；
3. 实际转账金额必须与订单金额一致；
4. 付款方名称必须与测试企业名称一致；
5. 同一笔流水未被其他订单使用。

常见需要走人工确认的情况包括：

1. 自动识别过程中未能正确匹配到对应入账记录；
2. 系统未及时完成自动处理，但转账信息本身无误；
3. 需要人工补充核对到账情况的其他场景。

进入兜底流程后，应按原有人工确认方式继续处理，由人工核对转账信息并完成后续确认。

以下情况应视为异常场景，不应作为人工兜底通过的前提：

1. 订单创建时间晚于实际转账时间；
2. 转账金额与订单金额不一致；
3. 付款方名称与测试企业名称不一致；
4. 收款账户不是指定测试账户；
5. 同一笔流水已被其他订单使用。

## 5. 配置 hosts

### 5.1 需要添加的映射

测试机需要添加以下 `hosts`：

```text
202.105.111.62  fbpop.uat.cmbchina.cn
202.105.111.63  site.cmbchina.com
202.105.111.63  uatub.cmbchina.com
```

说明：

- 以上端口均为 `80`；
- 这些配置仅用于测试环境；
- 测试结束或切换生产环境前，请删除这些映射。

### 5.2 Windows

编辑文件：

`C:\Windows\System32\drivers\etc\hosts`

建议：

1. 使用“记事本”或其他编辑器以管理员身份打开；
2. 追加上述三行配置后保存；
3. 打开命令提示符执行 `ipconfig /flushdns` 刷新 DNS。

### 5.3 macOS

编辑文件：

`/etc/hosts`

可执行：

```bash
sudo vi /etc/hosts
```

或：

```bash
sudo nano /etc/hosts
```

保存后执行：

```bash
sudo dscacheutil -flushcache
sudo killall -HUP mDNSResponder
```

### 5.4 Linux

编辑文件：

`/etc/hosts`

可执行：

```bash
sudo vi /etc/hosts
```

或：

```bash
sudo nano /etc/hosts
```

如系统启用了 `systemd-resolved`，执行：

```bash
sudo resolvectl flush-caches
```

如果未启用，可直接重新打开浏览器，或按实际发行版方式刷新 DNS。

## 8. 自动充值测试步骤

### 8.1 系统侧准备订单

1. 在系统中创建一笔充值订单；
2. 充值金额建议使用易识别的小额整数，例如 `1.00`、`2.00`、`5.00`；
3. 记录订单号、充值金额、创建时间；
4. 确认企业名称已显示为 `企业网银新20161090`；
5. 创建完成后再进入网银操作，不要先转账后补建订单。

### 8.2 申请账户发起转账

1. 使用任一测试网银用户登录测试网银并发起转账申请；
2. 菜单路径：`转账支付` -> `转账` -> `单笔转账经办（新）`；
3. 选择付款账号 `755915680110206`；
4. 发起转账，收款账号填写 `755915680110101`；
5. 收款账户必须填写为 `755915680110101`，不可改为其他账户；
6. 收款户名填写 `企业网银新20161090`；
7. 转账金额必须与系统订单金额完全一致；
8. 付款方名称应与测试企业名称保持一致；
9. 摘要、用途等字段建议填写与订单号相关的信息，便于排查；
10. 请确保本次转账时间晚于订单创建时间；
11. 提交转账申请，等待另一网银用户审批。

### 8.3 主账户审批转账

1. 退出当前网银，使用另一测试网银用户登录；
2. 菜单路径：`首页` -> `我的代办`；
3. 找到对应条目并进入审批；
4. 找到上一步由 `755915680110206` 发起、转入 `755915680110101` 的转账申请；
5. 重点核对收款账户是否为 `755915680110101`，并确认金额、户名、账号无误；
6. 审批成功后，等待银行侧生成收款流水。

### 8.4 上传出账回单（触发自动充值）

转账审批完成后，需要由测试人员主动下载出账回单并上传到系统，系统才会触发自动充值识别流程。

操作要求：

1. 下载出账回单：见下方 `8.5 电子回单下载`；
2. 将出账回单上传至系统对应充值订单。

### 8.5 系统自动识别充值

审批成功后，请在系统中观察本笔充值订单的处理结果。

本次测试应重点确认以下现象：

1. 订单状态能够从待处理逐步变为充值成功；
2. 到账金额与订单金额一致；
3. 本笔订单未出现重复充值；
4. 如系统页面提供流水号、回单或到账记录展示，应能看到对应结果；
5. 如系统支持查看处理日志或处理记录，应能看到本次充值已被正常处理。

如果系统已有定时任务处理银行入账，请按实际任务周期等待结果；如果系统支持手动触发处理，请由相关人员执行一次处理任务后再观察订单状态。

### 8.6 电子回单下载

当需要人工核对或留存回单时，可按以下步骤下载电子回单：

1. 菜单路径：`账务管理` -> `账务查询` -> `交易查询`；
2. 选择付款账号；
3. 选择交易日期等查询条件并点击查询；
4. 选择对应交易记录；
5. 点击打印回单，下载电子回单文件。

## 9. 验收要点

满足以下结果即可判定本次测试通过：

1. 网银经办与审批链路可正常执行；
2. `755915680110206 -> 755915680110101` 的转账成功入账；
3. 收款账户始终为 `755915680110101`；
4. 订单创建时间早于实际转账时间；
5. 付款方名称与测试企业名称一致；
6. 系统能够查询到该笔收款流水；
7. 系统能够定位到对应流水号；
8. 系统能够下载或获取该笔回单；
9. 充值订单自动变更为成功状态；
10. 同一流水不会被重复核销。

除自动充值成功场景外，还应补充验证以下兜底场景：

1. 转账信息本身无误，但系统未自动完成充值时，可继续按原有人工确认流程处理；
2. 自动识别未命中，但人工核对确认信息一致时，可继续人工确认；
3. 故意制造异常数据时，系统不应自动充值，人工确认也应保持同样的校验标准，不应放宽。

## 10. 常见问题排查

### 10.1 网银页面打不开

优先检查：

1. `hosts` 是否已正确配置；
2. DNS 缓存是否已刷新；
3. 是否能连通 `fbpop.uat.cmbchina.cn` 和 `cdctest.cmburl.cn:80`；
4. 是否使用了支持的浏览器。

### 10.2 系统查不到流水

优先检查：

1. 付款是否已经审批成功；
2. 收款账户是否填写为 `755915680110101`；
3. 查询时间范围是否覆盖本次交易时间；
4. 订单创建时间是否早于实际转账时间；
5. 是否命中了接口限频。

注意：

- 账务查询、交易管家查询接口，同一账号两次查询间隔应至少 `10` 秒；
- 尽量避免整点、半点以及每 `10` 分钟的高峰时段触发定时查询。

### 10.3 自动充值未完成

优先检查：

1. 系统企业名称是否已改成 `企业网银新20161090`；
2. 订单金额和实际转账金额是否完全一致；
3. 收款账户是否为 `755915680110101`；
4. 付款方名称是否与测试企业名称一致；
5. 订单创建时间是否早于实际转账时间；
6. 系统是否成功识别到对应收款流水和回单；
7. 同一流水是否已被其他订单核销。

如果最终转为人工确认，也应继续按上述条件核对，不应因自动失败而跳过时间、金额、企业名称或流水唯一性校验。

### 10.4 先转账后创建订单

这是本次测试需要重点关注的场景。

如果先完成银行转账，后创建充值订单，系统可能因时间顺序不满足匹配条件而无法自动完成充值。因此测试时必须严格按“先创建订单，再进行转账”的顺序执行。

## 11. 注意事项

1. 本文档面向业务测试人员，重点关注网银经办、审批和充值结果验证；
2. 测试环境仅用于联调和验收，请勿在生产环境进行测试；
3. 若系统存在定时拉取银行流水的任务，请尽量避开整点、半点和高峰时段观察结果；
4. 测试完成后请移除本机 `hosts` 中的测试映射。
