# 企银支付单笔经办 BB1PAYOP

> 来源: <https://openbiz.cmbchina.com/clouddc-document?bizkey=DCCT20201214175701770&subclass=1&treeID=2026032618001049277>
>
> 更新时间：2026-03-27 19:55:13 | 版本号：1.0.0

## 使用场景

该接口用于发起单笔支付经办。

## 使用说明

1. 支持客户在企业网银上设置审批流程，如果设置了审批流程，则发起经办后，需要管理员在企业网银上进行审批，才能对外发起支付。

2. 目前该支付API支持普通模式（100万以下走小额支付系统，100万以上走大额支付系统）、快速模式（100万以下走小额支付系统，100万以上走大额支付系统）、实时模式（走超网渠道，限额100万）、智能路由模式（根据输入的收方信息择优选择人行大额/小额/超级网银/同行支付系统进行汇出）。

   **结算方式选择 "R 实时"：**
   - 主要支持跨行
   - 收方为非招行账户，且为标准银联卡，则转账最少要素为：收方账号+收方户名
   - 收方为非招行账户，且为非标准银联卡（如对公账户等），则转账最少要素为：收方账号+收方户名+收方标准银行总行名（如中国银行）/总行行号

   **结算方式选择 "G 普通"、"Q 快速"：**
   - 收方为招行账户（含个人及对公账户），则转账最少要素为：收方账号+收方户名
   - 收方为非招行账户，则转账最少要素为：收方账号+收方户名+收方行名/行号

   **结算方式选择 "I 智能路由"：**
   - 客户需输入账号、户名、行名或行号，其中行名或行号无需区分总行或支行（标准银联卡只需输入账号、户名）
   - 若收方开户行支持超网入账，则优先通过超网渠道进行汇出，保障收方入账效率
   - 当选择非直汇模式（drpFlg字段传A）时，若未输入行号且输入行名不规范，则交易可能落地人工处理，影响入账时效
   - 智能路由仅用于辅助识别汇路，请尽可能完善支付要素，提升支付成功率

   > 注：若客户未填写开户地，计费将根据行名/行号判断开户地进行同城异地判断，若行名中不含明确的地址信息，可能存在同城异地的误判，请知悉。

3. 支付经办接口返回的经办成功结果仅代表"资金提出成功"，到账还是需要对手行进行入账处理，请调用 **企银支付业务查询 BB1PAYQR** 接口查询支付的最终状态。对于入账效率方面，每个银行的入账效率是不统一的，有个别城商行农信社入账需要人工落地处理，导致入账失败退款时间较长，可能要接近日终才会展现，因此建议客户可日终通过查询退票接口进行查询退票信息。

4. 单笔支付经办接口支持异步应答机制，银行可主动向客户推送支付结果（含退票结果）。

5. 请求报文字段 `yurRef`（业务参考号）用于标识该笔业务的编号，企业银行编号+业务类型+业务参考号必须唯一。企业自定义业务参考号，相同业务的业务参考号要始终保持唯一。注意：支付业务参考号仅保留最近13个月的交易数据校验，超过13个月的业务参考号无法校验重复性。

6. 由于失败的业务参考号可以重用，因此当用户使用同一业务参考号发送失败多次后，有可能会产生相同业务参考号的多笔记录。用户可以按照 **业务参考号+流程实例号**（同步返回的信息 `reqNbr`）作为唯一键值匹配支付结果，或者用户可以通过时间顺序判断最近一次经办的结果（同一天的支付业务，最大的流程实例号为最近一次经办的结果）。

7. 当企银支付业务查询（BB1PAYQR）响应请求状态 `reqSts` 和业务处理结果 `rtnFlg` 是 **FIN+【F/B/R/D/C】** 时，表明客户发起的这笔支付已经是终态，且该笔款项没有成功支付到收款人账户。客户可以就该笔款项重新发起支付，重发时不能改变业务参考号，非上述状态均不得重新发起支付。

8. 当返回报文 head 节点中 `resultcode` 字段为 `SUC0000` 时，表明请求已经被银行成功接收，不代表业务处理成功，业务处理结果请调用各个业务查询接口进行判断。返回其他错误码时，请先调用相应业务的查询接口判断之前请求是否有成功送达银行，确认没有记录后可以重发之前的请求。请注意：重发时请保证请求报文里面的业务参考号字段和之前请求里面的业务参考号一致（有特殊说明的接口除外），否则存在重复提交的风险。

9. 支持记账子单元。

10. 如果客户同时输入收方的行名和行号，则以行号为准进行汇出。

11. 若经办接口返回 FIN+F，可以判定为经办失败，无需再次调用查询接口，其他情况需调用业务查询接口确认最终结果。
